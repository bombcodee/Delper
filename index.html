용어집

프로세스 상세 가이드

각 개발 단계에서 무엇을 하고, 어떻게 하고, 무엇을 주의해야 하는지를 실전 수준으로 정리했습니다.

💡 1. Ideation — 아이디어 발상

▶

이 단계의 목표: "풀 가치가 있는 문제"를 찾고, 한 문장으로 정의하는 것.

실전 방법

1. Problem Statement 작성법: "[타겟 사용자]는 [상황]에서 [문제]를 겪고 있다." 형식으로 작성. 예: "ADHD 성향의 사람들은 비구조화된 생각을 정리하는 데 과도한 인지 부하를 겪고 있다."

2. Value Proposition 작성법: "[제품]은 [타겟]이 [문제]를 해결할 수 있게 [핵심 가치]를 제공한다." 형식. Problem Statement의 답이 되어야 함.

3. Feasibility Study: 세 가지를 확인 — ① 기술적으로 만들 수 있는가? (AI API 존재 여부, 핵심 기능 구현 가능성) ② 비용이 감당 가능한가? (AI API 월 예상 비용, 서버 비용) ③ 시장이 존재하는가? (검색량, 커뮤니티 반응)

산출물 체크리스트

  • Problem Statement 한 문장 작성 완료
  • Value Proposition 한 문장 작성 완료
  • 기술적 가능성 확인 (핵심 API/기술 존재 여부)
  • 월 예상 비용 대략 계산 (서버 + API + 도메인)
  • 비슷한 제품 3개 이상 검색 완료

통과 기준

Problem Statement와 Value Proposition이 명확하고, 기술적으로 구현 가능하며, 비용이 런웨이 내에서 감당 가능하면 다음 단계로.

⚠️ 실패 패턴

"이거 대박이야!" 확신만으로 검증 없이 바로 코딩 시작. 아이디어에 반했을 때가 가장 위험. 최소 1~2일은 냉각기를 두고 객관적으로 재검토.

🔍 2. Discovery — 시장·경쟁·사용자 조사

▶

이 단계의 목표: "이 제품을 만들 가치가 있는가?"를 데이터로 확인하는 것. 실제 0순위 단계.

실전 방법 (이 순서대로)

1단계: Competitive Analysis (경쟁 분석)
→ 비슷한 제품 5~10개 찾기. 각각의 핵심 기능, 가격, 타겟, 장단점을 표로 정리.
→ 핵심 질문: "이미 있는데 왜 또 만드는가?" — 답이 명확하지 않으면 차별점이 없는 것.
→ 도구: Google, Product Hunt, 디스콰이엇, 앱스토어 검색

2단계: User Persona 정의
→ 1~2개만. 너무 많으면 초점이 흐려짐.
→ 형식: 이름, 나이, 직업, 핵심 고민, 현재 해결 방법, 불만족 포인트.
→ 예: "민수, 28세, 프리랜서 개발자, ADHD 진단, 노션 쓰다 포기, 생각이 정리 안 돼서 프로젝트 시작을 못 함"

3단계: TAM/SAM/SOM 시장 규모
→ TAM(전체 시장): 전 세계 노트/생산성 앱 시장
→ SAM(접근 가능): 한국어 사용자 중 AI 노트앱에 관심 있는 사람
→ SOM(실제 확보 가능): 첫 6개월 내 현실적으로 확보할 수 있는 유료 사용자 수
→ 정확한 숫자보다 "이 시장이 충분히 큰가?"의 감을 잡는 게 목적

4단계: User Interview / Survey
→ 최소 5명. 타겟 사용자에게 직접 물어봐야 함.
→ The Mom Test 원칙: "이런 앱 만들면 쓸 거예요?"는 나쁜 질문 (다들 "네"라고 답함). "지금 이 문제를 어떻게 해결하고 있어요?" "마지막으로 이 문제 때문에 고생한 게 언제예요?"가 좋은 질문.
→ 채널: Reddit, 커뮤니티, 카카오톡 오픈채팅, 지인

산출물 체크리스트

  • 경쟁 제품 비교표 작성 (5개 이상)
  • 차별화 포인트 한 문장 정리
  • User Persona 1~2개 작성
  • TAM/SAM/SOM 대략적 감 잡기
  • User Interview 최소 5명 완료
  • 인터뷰에서 발견한 핵심 인사이트 3개 이상 정리

통과 기준

인터뷰 대상 5명 중 3명 이상이 "이 문제를 실제로 겪고 있다"고 답하고, 현재 대안에 불만족하며, 차별화 포인트가 명확하면 다음 단계로.

⚠️ 실패 패턴

Discovery를 건너뛰고 바로 Planning으로 가는 것. "조사할 시간에 만드는 게 빠르다"는 착각. 이 단계를 1주만 해도 6개월간 잘못된 방향으로 만드는 것을 막을 수 있음.

📋 3. Planning — 기획

▶

이 단계의 목표: "뭘 어떻게 만들 건가"를 문서로 확정하는 것. 바이브코딩에서 AI에게 줄 컨텍스트의 품질이 여기서 결정됨.

PRD 작성법 (1인 개발 최적화)

PRD 하나에 BRD + AC를 통합. 구조:

1. 왜 만드는가 (BRD 영역) — Problem Statement, Value Proposition, 타겟 사용자, 성공 지표
2. 뭘 만드는가 — 기능 목록을 MoSCoW로 분류
  → Must: 이게 없으면 제품이 아님 (MVP 핵심)
  → Should: 있으면 훨씬 좋지만 v1에 없어도 됨
  → Could: 여유 있으면 넣을 것
  → Won't: 이번에는 명시적으로 안 함
3. 각 기능별 AC — "정리 버튼 클릭 → 10초 내 결과 표시" 같은 완료 기준
4. Edge Case — "빈 텍스트 입력 시?", "10만 글자 입력 시?", "네트워크 끊김 시?"

User Flow 작성법

핵심 시나리오 3~5개만. 모든 경로를 그리려 하면 끝이 없음.
→ 시나리오 1: 신규 가입 → 온보딩 → 첫 노트 작성 → 정리 요청 → 결과 확인
→ 시나리오 2: 기존 사용자 → 로그인 → 이전 노트 열기 → 추가 작성 → 정리
→ 시나리오 3: 무료→유료 전환 흐름
도구: Excalidraw(무료), FigJam, 종이+펜도 충분.

Wireframe 실전 팁

User Flow의 각 화면을 스케치. 예쁠 필요 없고 요소 배치만.
→ 한 화면에 핵심 행동(CTA)이 하나만 있는지 확인
→ 모바일 화면도 같이 그려야 반응형 설계를 미리 고려할 수 있음

Tech Spec 작성법

→ 기술 스택 확정 (프론트, 백엔드, DB, 배포, 인증, 결제)
→ DB 스키마 초안 (ERD) — 최소한 핵심 테이블과 관계
→ API 엔드포인트 목록 — 각 기능에 필요한 API 정리
→ 외부 서비스 연동 목록 (AI API, 인증, 결제, 파일 저장소)
→ 인증 방식 결정 (이메일/소셜 로그인, JWT/세션)

Estimation (공수 산정) 실전법

→ Must 기능 각각에 예상 시간 할당
→ 자기가 예상한 시간에 ×2를 곱하기 (경험적으로 항상 2배 걸림)
→ 전체 합이 4주 이내면 적정 MVP. 6주 넘으면 Must를 더 줄여야 함
→ 2주 스프린트 단위로 쪼개서 "1차 스프린트에서 뭘 할지"까지 결정

산출물 체크리스트

  • PRD 작성 완료 (BRD + 기능 MoSCoW + AC + Edge Case)
  • User Flow 핵심 시나리오 3~5개
  • Wireframe 주요 화면
  • Tech Spec (스택 + DB 스키마 + API 목록)
  • 외부 서비스 연동 목록
  • MVP Definition (Must만 추출)
  • Estimation 완료 (×2 적용, 4주 이내 확인)
  • 1차 스프린트 범위 확정

⚠️ 실패 패턴: Feature Creep

MVP에 "이것도 있으면 좋겠다"를 계속 추가하다 영원히 출시 못 함. Won't 목록에 넣은 건 진짜로 안 만들어야 함. Must에 있는 기능도 의심하기 — "이거 없으면 진짜 제품이 안 되나?"

💻 4. Development — 코딩

▶

이 단계의 목표: Planning의 산출물에 따라 동작하는 코드를 만드는 것. Sprint 단위로 쪼개서 2주마다 동작하는 결과물을 만들기.

개발 시작 전 환경 세팅

  • GitHub 레포 생성 + .gitignore 설정
  • 브랜치 전략 결정 — 1인 개발: main + feature 브랜치면 충분
  • 커밋 컨벤션 설정 — feat: / fix: / refactor: / docs: 형식
  • Linting/Formatting 설정 (ESLint + Prettier)
  • TypeScript 설정 (규모가 커질 제품이면)
  • 환경변수(.env) 설정 — .env.local / .env.production 분리
  • CI/CD 설정 — push하면 자동 배포 (Vercel이면 자동)
  • CLAUDE.md 작성 — 프로젝트 구조, 기술 스택, 코딩 규칙 정리
  • AI Mock 데이터 준비 — 개발 중 실제 API 호출 비용 절감

Sprint 운영법 (1인 개발 버전)

→ 1 Sprint = 2주. 매 스프린트 시작 시 "이번 2주에 뭘 만들지" 3~5개 태스크 선정.
→ 각 태스크는 하루 안에 끝낼 수 있는 크기로 쪼개기. "로그인 기능"이 아니라 "소셜 로그인 버튼 UI", "Google OAuth 연동", "로그인 후 리다이렉트" 같은 단위.
→ Kanban 보드 (Notion/Linear)에 Todo → In Progress → Done으로 관리.
→ 스프린트 끝날 때 동작하는 결과물이 있어야 함. 안 되면 스코프를 줄인 것.

바이브코딩 실전 팁

→ AI에게 지시할 때 PRD의 해당 기능 + AC + Tech Spec을 같이 줘야 일관된 코드가 나옴
→ "이거 만들어줘"보다 "User Flow 3단계의 화면을 만들어줘. Tech Spec의 API 구조를 따르고, AC는 이거야"가 10배 나은 결과물을 줌
→ AI가 만든 코드는 반드시 읽고 이해한 후 커밋. 이해 안 되면 설명 요청.
→ 하루 끝에 10분간 Code Review 시간 — "오늘 만든 코드에서 나중에 문제 될 것" 체크

Refactoring 타이밍

→ "나중에 고치지" 하면 영원히 안 고침. Technical Debt를 의식적으로 추적.
→ 규칙: 같은 코드를 3번 복붙하면 그때 리팩토링. 그전에는 중복이 아직 패턴이 안 된 것.
→ 매 3번째 스프린트는 새 기능 대신 리팩토링 스프린트로. 기술 부채를 주기적으로 갚기.

개발 순서 (이 순서가 중요)

① 인증 (로그인/회원가입) — 모든 기능의 기반
② DB 스키마 + 핵심 CRUD — 데이터 구조가 잡혀야 화면을 만들 수 있음
③ 핵심 기능 1개 — 제품의 존재 이유. SWAI면 "텍스트 입력 → AI 정리"
④ 기본 UI/레이아웃 — 네비게이션, 사이드바, 기본 페이지 구조
⑤ 나머지 Must 기능들
⑥ 결제/구독 (유료 제품이면)

✅ 5. QA — 테스트·검증

▶

이 단계의 목표: 만든 게 제대로 동작하는지 확인. 1인 개발자가 현실적으로 할 수 있는 수준으로.

테스트 피라미드 (1인 개발 현실 버전)

교과서적인 테스트 피라미드는 Unit → Integration → E2E 순서지만, 1인 개발에서는 현실적으로:

반드시 해야 할 것:
→ Smoke Test: 배포 후 핵심 기능 3~5개 수동 확인. "로그인 되나? 노트 만들어지나? 정리 동작하나?"
→ Unit Test: 핵심 비즈니스 로직에만. 모든 함수에 테스트를 쓸 필요 없음. "돈이나 데이터 무결성에 관련된 로직"에만.
→ Cross-browser: 크롬 + 사파리 + 모바일 최소 확인. 실제 디바이스로.

하면 좋은 것:
→ UAT: 주변 사람 2~3명에게 "이거 써봐"하고 옆에서 지켜보기. 어디서 멈추는지, 뭘 헷갈려하는지 관찰.
→ Integration Test: API 엔드포인트별 요청/응답 확인.

나중에 해도 되는 것:
→ E2E Test (Playwright/Cypress): 기능이 10개 이상으로 늘어났을 때.
→ Load Test: 사용자 1000명 이상 될 때.
→ Regression Test 자동화: 릴리즈 주기가 빨라졌을 때.

테스트 판단 기준

"이 기능이 깨지면 사용자가 돈을 잃거나, 데이터를 잃거나, 서비스를 못 쓰는가?" — Yes면 테스트 필수. No면 수동 확인으로 충분.

QA 체크리스트

  • 핵심 기능 Smoke Test 통과
  • 결제/데이터 관련 로직 Unit Test 작성
  • Chrome + Safari + 모바일 기본 확인
  • 주변 2~3명 UAT 완료
  • 에러 발생 시 사용자에게 보이는 화면 확인 (빈 화면 아닌지)
  • 네트워크 끊김/느린 환경 테스트

🚀 6. Launch — 배포·출시

▶

이 단계의 목표: 제품을 세상에 내놓고, 초기 사용자를 확보하는 것.

Launch 전략 선택

옵션 A: Closed Beta (초대제) — 10~50명에게만 공개. 피드백 밀도 높음. "초대받은" 느낌으로 충성도 형성. Waitlist로 기대감 조성.
옵션 B: Open Beta — 누구나 가입 가능하되 "Beta" 딱지 붙이기. 버그에 대한 관용 확보. 빠른 사용자 확보.
옵션 C: 정식 출시 — 완성도 높을 때. Product Hunt / 디스콰이엇 런치와 동시에.

1인 개발자 추천: 옵션 A → B → C 순서. Closed Beta로 핵심 피드백 받고, Open Beta로 확장하고, 정식 출시.

Product Hunt / 디스콰이엇 런치 준비

→ 런치 2주 전: 랜딩 페이지 완성, OG 태그 설정, 스크린샷/GIF 준비, 한 줄 설명 확정
→ 런치 1주 전: 커뮤니티에 미리 공유 (빌드 인 퍼블릭), 지인에게 런치 당일 업보트 요청
→ 런치 당일: 오전 9시(PST) 게시, 댓글에 실시간 응답, SNS에 공유
→ 런치 후: 방문자 → 가입 전환율 실시간 추적, 피드백 즉시 반영

GTM (Go-To-Market) 실전

→ 채널 1순위: 타겟 사용자가 이미 모여있는 곳 (ADHD 커뮤니티, 생산성 포럼, 개발자 커뮤니티)
→ 채널 2순위: SEO (블로그 글 3~5개 미리 작성해서 런치와 동시에 게시)
→ 채널 3순위: 소셜 미디어 (트위터/X에 빌드 과정 공유해왔다면 런치 때 팔로워가 초기 사용자)

Launch 체크리스트

  • HTTPS 적용 확인
  • 환경변수 production 설정 확인
  • 에러 트래킹 연동 (Sentry)
  • Analytics 연동 (Google Analytics / PostHog)
  • 개인정보처리방침 게시
  • 이용약관 게시
  • 사업자 정보 하단 표시
  • OG 태그 / Social Preview 설정 + 카카오톡 공유 테스트
  • Google Search Console 등록
  • Rollback Plan 준비 (문제 시 이전 버전 복구 방법)
  • Beta/Launch 전략 결정 + 채널 확정
  • 랜딩 페이지 히어로 섹션 검증 (3초 안에 가치 전달?)
  • 온보딩 흐름 확인 (가입→첫 가치 경험까지 마찰 없는지)

🔄 7. Iteration — 피드백 기반 개선

▶

이 단계의 목표: 사용자 데이터와 피드백을 기반으로 제품을 반복 개선. 4단계(Development)로 회귀하여 순환.

피드백 수집 방법

→ 정량 데이터: Analytics (어디서 이탈하는지), Funnel 분석 (가입→활성화→결제 전환율)
→ 정성 데이터: 사용자 인터뷰, 인앱 피드백 버튼, 이메일 응답, 디스코드/커뮤니티 의견
→ 세션 녹화: Hotjar/Clarity로 사용자가 실제로 어떻게 쓰는지 관찰
→ NPS 설문: 가입 7일 후 / 30일 후에 자동 발송

우선순위 결정 프레임워크

"다음에 뭘 할까?"를 결정하는 게 Iteration의 핵심. 이 매트릭스로 판단:

높은 영향 · 낮은 노력

  • 즉시 실행. Quick Win.
  • 버그 수정, 작은 UX 개선

높은 영향 · 높은 노력

  • 다음 스프린트에 계획.
  • 새 핵심 기능, 큰 리팩토링

낮은 영향 · 낮은 노력

  • 여유 있을 때.
  • 사소한 디자인 개선

낮은 영향 · 높은 노력

  • 하지 않음.
  • 과도한 기능, 오버엔지니어링

새 기능 vs 버그 수정 vs 기술 부채

→ 경험적 최적 비율: 새 기능 60% + 버그 수정 20% + 기술 부채 20%
→ 사용자에게 영향을 주는 버그는 새 기능보다 항상 우선
→ 매 3번째 스프린트는 기술 부채 정리 스프린트로 예약
→ "이 기술 부채를 지금 안 갚으면 3개월 후에 어떻게 되는가?"로 긴급도 판단

Retrospective (회고) 실전법

→ 매 스프린트(2주) 끝에 15분만 투자.
→ 3가지만 적기: ① 잘된 것 ② 안된 것 ③ 다음에 바꿀 것
→ 제품 회고가 아니라 프로세스 회고. "코드를 어떻게 고칠까"가 아니라 "일하는 방식을 어떻게 고칠까"

PMF (Product-Market Fit) 판단 기준

→ Day 30 잔존율 20% 이상이면 PMF에 가까움
→ "이 제품이 없어지면 매우 실망한다"고 답하는 사용자가 40% 이상이면 PMF
→ 3개월간 핵심 지표(North Star Metric)가 0성장이면 피벗 검토
→ 다만 초기 3개월은 자연스럽게 느림. 최소 6개월은 지켜봐야 함

Iteration 체크리스트

  • Analytics 데이터 주간 리뷰
  • 사용자 피드백 수집 채널 운영 중
  • 피드백을 영향/노력 매트릭스로 분류
  • Retrospective 2주마다 실행
  • Technical Debt 목록 관리 중
  • North Star Metric 정의 및 추적 중
  • Changelog 업데이트

⚡ Activation (활성화) — "아하 모먼트"에 도달했는가? → Day 1 잔존율, Time to Value
🔄 Retention (유지) — 계속 돌아오는가? → Day 7/30 잔존율, DAU/MAU

핵심 지표 세부 설명

⭐ North Star Metric

제품 핵심 가치를 대표하는 단 하나의 지표. SWAI면 "주당 정리된 노트 수". 이게 올라가면 다른 지표도 따라옴.

👥 DAU / MAU

일간/월간 활성 사용자. 사용자가 실제로 쓰고 있는지의 기본 지표. DAU/MAU 비율이 높을수록 습관화된 제품.

💵 MRR / ARR

월간/연간 반복 수익. SaaS의 기본 건강 지표. MRR = 유료 사용자 수 × 평균 결제 금액.

💎 LTV

Customer Lifetime Value. 고객 한 명이 전체 기간 동안 가져다주는 총 수익. LTV = ARPU ÷ 월간 이탈률.

💸 CAC

Customer Acquisition Cost. 고객 한 명 획득 비용. LTV/CAC ≥ 3이어야 건강. <1이면 돈 쓸수록 손해.

🔥 Burn Rate / Runway

월간 지출 속도와 남은 자금으로 버틸 수 있는 기간. Runway = 잔고 ÷ 월 지출. 6개월 미만이면 MVP 범위 축소.

▶

번아웃

주 50시간 이상 → 생산성 하락. 의식적 휴식 계획 필요.

버스 팩터 = 1

문서화, 비밀번호 관리, 자동화된 배포로 리스크 완화.

의존성 리스크

특정 서비스 100% 의존 → 대체 가능한 설계 필요.

기술 락인

추상화 레이어 또는 표준 기술(PostgreSQL 등) 사용.

규제 변경

AI 규제, 플랫폼 정책 변경. 단일 플랫폼 의존은 사업 리스크.

데이터 유출

72시간 내 신고 의무. 대응 계획 사전 수립 필수.

수익 다각화 실패

수익원이 하나면 막히면 끝. 구독 + 원타임 + 어필리에이트 + API 조합.

법적 분쟁 대비 부재

이용약관에 면책 조항, 서비스 중단 권리, 환불 정책 명확히. AI 서비스는 "잘못된 정보" 면책 필수.

경쟁자 진입

대기업이 같은 기능 무료 출시하면? "기능"이면 죽고 "커뮤니티/특화 경험"이면 살아남음.

환율/국제 정세

해외 결제 수익은 환율 변동에 직접 영향. 달러 약세 시 원화 환산 수익 감소.

⚖️ 법적/사업 필수 사항

▶
  • 사업자등록 + 통신판매업 신고 — 유료 서비스 운영 시 법적 필수
  • 개인정보처리방침 게시 — 이름, 이메일만 받아도 필수
  • 전자상거래법 표시 — 사업자 정보, 환불 규정 하단 표시
  • PG사 심사 — 1~2주 소요, 개발 완료 전 신청 권장
  • 상표 등록 — 키프리스 검색 후 초기 등록 권장
  • 세금/회계 — 삼쩜삼, 자비스 활용. 해외 수익은 외화 신고 별도
  • AI 콘텐츠 저작권 — 법적으로 불명확한 영역, 리스크 인식 필요

💵 비용 최적화

▶

무료 티어 활용 (월 $0 시작): Vercel(무료) + Supabase(무료) + Cloudflare(무료) + GitHub(무료)

AI API 비용 절감: 캐싱 · 요청 배칭 · 모델 티어 분리 (간단한 요청은 저렴한 모델, 복잡한 요청만 고급 모델)

CDN/이미지 최적화: Cloudflare Images, Vercel Image Optimization 활용. 이미지 트래픽이 서버비의 주범인 경우가 많음. WebP/AVIF 포맷 변환, lazy loading 적용.

서버리스 vs VPS 전환 시점: 트래픽이 적을 때(월 100만 요청 이하)는 서버리스가 압도적으로 저렴. 일정 규모 넘으면(월 $50+ 서버리스 비용) VPS($5~10/월)가 더 싸져. 이 전환 시점을 인식해야 해.

수익 보호: Stripe Radar(부정결제 방어) · 동시접속 제한 · API 키 남용 방지 · 크롤링 방어

런웨이(남은 개월 수)를 명확히 계산하면 불안이 줄어듦.

80%면 출시. "Done is better than perfect."

🎯 스코프 불안

📚 학습 vs 실행

Just-in-time Learning: 지금 제품에 필요한 것만 즉시 배우기.

🧩 멀티태스킹 함정

시간 블록킹: 오전은 개발만, 오후 1시간은 마케팅만.

🏁 성공 정의

자기만의 기준 없으면 남의 기준으로 자기를 평가하게 됨.

🔄 피벗 판단

3개월간 성장 지표 0이면 PMF 의심. 다만 초기 3개월은 느린 게 정상, 최소 6개월 관찰.

🚫 실패 패턴 (Anti-patterns)

▶

Feature Creep

MVP에 기능을 계속 추가하다 영원히 출시 못 함. 가장 흔한 사망 원인.

Resume Driven Development

배우고 싶은 기술을 억지로 넣는 것. 익숙한 기술로 빠르게 만들기.

피드백 없는 개발

6개월간 아무에게도 안 보여주고 만들다가 아무도 안 쓰는 제품 출시.

가격 설정 실패

너무 싸게 시작하면 올리기 어려움. 처음부터 유료 + 무료 체험.

단일 채널 의존

트래픽 80%가 한 채널이면, 알고리즘 변경 한 번에 끝.

지표 무시

감으로 운영하면 문제의 원인을 알 수 없음. Analytics 필수.
